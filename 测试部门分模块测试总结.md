## 1. Agent 主循环与工具调用模块

涉及测试文件：

- `tests/test_agent.py`
- `tests/resilience/test_error_recovery.py`
- `tests/test_runtime.py`
- `tests/test_harness.py`
- `tests/trajectory/test_trajectory_evaluator.py`

测试手段：

- 使用 Mock LLM 客户端模拟模型流式输出、文本回复、工具调用、异常返回和多轮对话。
- 使用异步测试验证 Agent loop 的事件流、停止条件、并发工具执行和错误恢复。
- 使用声明式 scenario runner 验证工具调用轨迹和文件操作记录。
- 使用 trajectory evaluator 对工具选择、安全性、冗余操作、错误恢复等维度进行评分测试。

进行了什么测试：

- 单步工具调用：模型请求 `ReadFile`，Agent 执行工具并把结果回传。
- 多步自主执行：模型先 `WriteFile`，再 `ReadFile` 验证，最后输出结果。
- 停止条件：正常 `end_turn`、最大轮数、取消执行、连续未知工具终止。
- 工具调用分区：可并发工具与必须串行工具的调度顺序。
- Token 统计：输入/输出 token 能正确累计。
- Plan Mode：计划模式下拒绝写操作，普通模式下正常执行。
- 异常恢复：空响应、未知工具、缺失工具 ID、工具参数错误、工具执行失败后继续循环。
- 轨迹评分：正确工具链得高分，危险命令、冗余读取、错误恢复不足会被扣分。

证明了什么功能没有问题：

- Agent 可以稳定完成 ReAct 风格的“模型生成工具调用 -> 执行工具 -> 回填结果 -> 继续推理”循环。
- 多轮工具调用、并发调度、停止条件和 token 统计行为正确。
- 面对模型异常输出或工具执行错误时，系统不会直接崩溃，可以生成可追踪的错误事件并继续或安全终止。
- 轨迹评估机制可以识别安全风险、冗余操作和工具选择质量。

## 2. 权限、安全与命令风险控制模块

涉及测试文件：

- `tests/test_permissions.py`
- `tests/security/test_command_injection.py`

测试手段：

- 参数化安全用例覆盖多种危险命令变体。
- 使用临时目录和符号链接验证路径沙箱。
- 使用端到端 Agent 测试验证权限拦截后主循环是否继续。
- 使用规则文件模拟用户级、项目级、本地级权限规则。

进行了什么测试：

- 危险命令检测：`rm -rf /`、`mkfs`、`dd` 写设备、`chmod -R 777 /`、fork bomb、覆盖系统文件等。
- 混淆变体检测：大小写混合、空白字符、链式命令、`sudo`/`su` 提权、`bash -c`/`eval` 间接执行、`curl | bash` 和 `wget | sh`。
- 安全命令放行：`ls`、`git status`、`grep`、`find`、`python --version` 等不会被误判。
- 路径沙箱：阻止访问 `/etc/passwd`、`~/.ssh`、路径穿越和符号链接逃逸，同时允许项目内路径和临时目录。
- 权限模式：`default`、`acceptEdits`、`plan`、`bypassPermissions`、`custom`、`dontAsk`。
- 规则引擎：规则解析、匹配、优先级覆盖、同级后写优先。
- 端到端权限行为：危险命令被拒绝、越界路径被阻止、用户拒绝操作后 Agent 继续运行。

证明了什么功能没有问题：

- 高风险 shell 命令和路径逃逸可以被有效拦截。
- 常规安全命令不会被过度拦截，误报风险可控。
- 多层权限模型能够按模式和规则正确决策。
- 权限拒绝不会破坏 Agent 主循环，系统能把拒绝结果反馈给模型。

## 3. 上下文压缩、恢复与大输出处理模块

涉及测试文件：

- `tests/test_context.py`
- `tests/perf/test_context_perf.py`
- `tests/test_recovery.py`
- `tests/test_replacement_state.py`

测试手段：

- 使用临时目录验证大工具输出落盘。
- 构造多条工具结果消息，验证单条限制、聚合限制和状态冻结。
- 使用性能测试验证 5MB、10MB 输出写入和 50/100 条工具结果处理速度。
- 构造恢复状态，验证 compact 后文件快照、技能快照和工具 schema 能重新附加。

进行了什么测试：

- 单条超大工具结果会写入磁盘，并在上下文里替换为 preview。
- 小结果保持原样，不会被不必要替换。
- 聚合超限时会选择部分 fresh 工具结果落盘，保证上下文总量受控。
- 已经持久化的结果不会重复处理。
- replacement state 不会修改原始 conversation，二次应用结果字节一致。
- 旧结果裁剪、compact 阈值、自动 compact 判断、summary 标签提取。
- RecoveryState 的文件数量限制、时间清理、单文件截断、技能预算控制。
- 性能要求：大输出持久化和大量小结果处理在可接受时间内完成。

证明了什么功能没有问题：

- 大型工具输出不会撑爆上下文窗口。
- 历史工具结果的替换决策稳定、可重放，不会污染原始对话对象。
- 自动压缩和恢复附件机制可以保留必要上下文，减少 compact 后遗忘关键文件/技能信息的风险。
- 大输出处理具备基本性能保障。

## 12. 总体结论

测试部门使用的主要手段包括：

- 单元测试：验证纯函数、解析器、数据模型和状态转换。
- 异步集成测试：验证 Agent loop、hook、任务管理和并发调度。
- Mock LLM/Mock HTTP：避免依赖真实外部服务，稳定复现模型和网络行为。
- 参数化安全测试：批量覆盖危险命令、混淆变体和安全命令误报。
- 临时文件和真实 git 仓库：验证文件系统、session、worktree 和 merge preview。
- 性能测试：验证大输出持久化和上下文压缩处理速度。
- 故障注入：验证空响应、未知工具、损坏 JSON、缺失工具结果、MCP 部分失败等异常场景。
- 轨迹评估：从工具链质量、安全性、冗余操作和错误恢复角度评估 Agent 行为。

通过 `608 passed` 可以证明：
